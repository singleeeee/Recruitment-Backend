# 社团管理员管理功能文档

## 概述

本文档描述了社团管理员管理功能的实现细节和API接口说明。

## 数据模型

社团和管理员之间的关系通过以下字段建立：

- `User` 表中的 `clubId` 字段：标识该用户管理的社团
- `Club` 表中的 `admins` 关系：包含所有管理该社团的用户

每个用户只能管理一个社团，确保权限清晰和责任明确。

## API 接口说明

### 1. 创建社团时指定管理员

**接口**: `POST /clubs`

**描述**: 创建新社团时可以可选地指定初始管理员

**请求体**:
```json
{
  "name": "计算机协会",
  "description": "这是一个专注于计算机技术的社团",
  "category": "科技",
  "logo": "https://example.com/logo.png",
  "adminIds": ["123e4567-e89b-12d3-a456-426614174000"]
}
```

**注意**: `adminIds` 字段是可选的，如果不提供，则创建的社团不会有任何管理员。

### 2. 更新社团管理员列表

**接口**: `PUT /clubs/:id/admins`

**描述**: 完全替换社团的管理员列表

**请求参数**:
- `id`: 社团ID (路径参数)

**请求体**:
```json
{
  "adminIds": ["123e4567-e89b-12d3-a456-426614174000", "123e4567-e89b-12d3-a456-426614174001"]
}
```

**效果**: 该操作会清除社团的所有现有管理员，然后添加新的管理员列表。

### 3. 添加社团管理员

**接口**: `POST /clubs/:id/admins`

**描述**: 为社团添加新的管理员

**请求参数**:
- `id`: 社团ID (路径参数)

**请求体**:
```json
{
  "adminId": "123e4567-e89b-12d3-a456-426614174002"
}
```

**效果**: 将指定的用户添加为该社团的管理员。如果该用户已经是其他社团的管理员，则操作会失败。

### 4. 移除社团管理员

**接口**: `DELETE /clubs/:id/admins/:adminId`

**描述**: 从社团移除指定的管理员

**请求参数**:
- `id`: 社团ID (路径参数)
- `adminId`: 要移除的管理员ID (路径参数)

**请求示例**: `DELETE /clubs/abc-123/admins/def-456`

**效果**: 将该用户从社团管理员列表中移除，该用户的 `clubId` 字段会被设置为 `null`。

## 业务规则和验证

### 1. 用户角色验证
- 只有角色为 `club_admin` 的用户才能被设置为社团管理员
- 如果尝试设置非社团管理员角色的用户，会抛出 `BadRequestException`

### 2. 唯一社团约束
- 每个用户只能管理一个社团
- 如果尝试将已管理其他社团的用户设置为新的社团管理员，会抛出 `BadRequestException`

### 3. 社团存在性验证
- 所有操作都会验证指定的社团是否存在
- 如果社团不存在，会抛出 `BadRequestException`

### 4. 用户存在性验证
- 所有操作都会验证指定的用户是否存在
- 如果用户不存在，会抛出 `BadRequestException`

## 错误处理

所有接口都可能抛出以下异常：

- `BadRequestException`: 请求参数不合法（用户不存在、社团不存在、用户角色错误等）
- `ConflictException`: 数据冲突（重复的操作等）
- `InternalServerErrorException`: 服务器内部错误（系统配置错误等）

## 实现细节

### 事务处理
管理员的添加、移除和批量更新操作都使用了数据库事务，确保数据一致性。

### 数据同步
所有管理员操作都会同时更新：
1. `User` 表中的 `clubId` 字段
2. `Club` 表中的 `admins` 关系

### 权限控制
所有社团管理员管理接口都需要超级管理员 (`super_admin`) 权限。

## 使用场景示例

### 场景1：创建新社团
1. 超级管理员创建社团，不指定管理员
2. 稍后通过 `POST /clubs/:id/admins` 添加第一个管理员

### 场景2：社团交接
1. 超级管理员使用 `PUT /clubs/:id/admins` 更新完全不同的管理员列表
2. 旧管理员会被自动从社团中移除，新管理员会被添加

### 场景3：管理员离职
1. 超级管理员使用 `DELETE /clubs/:id/admins/:adminId` 移除离职的管理员
2. 该管理员可以随时被分配到其他社团

## 安全的考虑

1. **输入验证**: 所有ID参数都使用UUID验证确保格式正确
2. **角色检查**: 确保只有正确的角色才能被设置为管理员
3. **事务保护**: 关键操作使用数据库事务保证数据一致性
4. **软删除**: 社团使用软删除策略，避免数据丢失
5. **权限隔离**: 只有超级管理员可以管理社团管理员分配
